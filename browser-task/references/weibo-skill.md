# 微博综合操作技能（子 Skill）

> 本文件是 GUI 浏览器自动化任务技能的子模块，当且仅当用户明确要在微博站内操作（登录 / 点赞 / 收藏 / 发布）或做微博舆情分析时被 `SKILL.md` 路由到。

本 Skill 涵盖两大功能模块，根据用户意图路由执行：

- **模块 A（GUI 基础操作）**：触发词——登录 / 点赞 / 收藏 / 发微博 / 发布内容 / 帮我操作微博
- **模块 B（舆情分析）**：触发词——微博舆情 / 热点 / 事件进展 / 检索 / 搜索 / 分析

两个模块可组合使用（例如"分析完后帮我发一条"先执行模块 B 生成文案，再路由到模块 A 执行发布）。

## 参考文件（导航）

- **约束与证据要求（触发模块 B 时每次必读）**：[weibo/CONSTRAINTS.md](weibo/CONSTRAINTS.md)
- **输出格式与交付模板（需要输出 Markdown 交付物或用户要求发帖/多文档时读取）**：[weibo/FORMAT.md](weibo/FORMAT.md)

## 模块 A：GUI 基础操作

本模块强制要求：所有操作在带界面的浏览器中完成，不使用任何非 GUI 接口或"假想已完成"的步骤。

### 输入要求（信息不足时的默认策略）

- **目标微博定位（点赞/收藏时）**：优先让用户提供微博条目的 URL；若用户不提供 URL，必须使用微博站内搜索先把目标微博搜索出来再进行后续操作（不要在信息流里盲目滚动寻找）。
- **发布内容（发布时）**：需要可直接发布的纯文本。若用户只说"帮我发一条"，先收集完整文本再进行 GUI 发布流程。
- **站内搜索入口（用于定位目标微博）**：微博站内搜索通常在左上角 Logo 旁边的放大镜图标；不要把关键词输入到信息流顶部的"发布用文本框/输入框"（该输入框用于发微博）。
  - **搜索关键词（点赞/收藏时）**：若用户未给 URL，你需要从用户处获得可搜索的定位信息（例如：发布人昵称、微博文案关键字、话题、发布时间大致范围）。信息不足时先向用户索取后再搜索。

### 通用执行原则（GUI 可靠性）

- **先确认登录态**：任何需要登录的动作（点赞/收藏/发布）开始前都要先检查是否已登录；未登录则先走登录流程。
- **可见性优先**：按钮/图标必须在可见区域内再点击；看不到就滚动或展开，直到目标元素清晰可见。
- **逐步验证**：每个动作点击后，必须观察 UI 反馈来确认成功（按钮高亮、数字变化、Toast 提示、菜单消失且状态更新等）。
- **避免误点**：同一页面可能有多个相似图标，必须以"目标微博条目"作为锚点，只在该条目内寻找对应按钮。
- **滚动工具**：优先使用 scroll 来滚动页面；避免拖拽滚动条和使用 PageDown/Home 等大步跳转方式。
- **慢速小步滚动**：滚动时要慢、幅度要小；微博条目可能较短，滚动过快会跳过目标元素或导致定位失败。
- **自适应缩小滚动幅度**：如果连续滚动多次仍未发现目标元素，把每次滚动幅度进一步减小，再继续慢速查找，直到目标元素进入可见区或确认当前页面确实不存在该元素。
- **遇到交互障碍就人工接管**：出现滑块、人机验证、短信验证码、扫码登录等交互，必须走人工协助流程（让用户手动处理并明确确认完成/失败），不得绕过或猜测结果。

### 登录流程（必须人工接管）

1. 在 GUI 浏览器中打开微博首页。
2. 在界面中寻找"登录"入口：
   - 优先找显式的"登录"文字按钮或头像/用户入口的"未登录"状态提示。
   - 若未找到，慢速小幅向下滚动后再次查找；若连续滚动几次仍找不到，进一步减小滚动幅度再查找，重复 2–3 次。
3. 找到登录入口后点击进入登录界面。
4. 立刻请求人工接管浏览器完成登录（使用 `AskHumanToControlBrowser`）。
5. 人工完成后，收回控制权并验证登录成功：
   - 页面右上角出现用户头像/昵称；或
   - 进入个人主页/消息等需要登录的入口不再跳转到登录页。
6. 若仍处于未登录状态，回到步骤 2 重试一次；仍失败则停止并清晰说明卡点（例如"验证码未通过/需要扫码/页面提示错误"）。

### 功能 A-1：点赞（登录后）

**目标**：对"目标微博条目"点击点赞按钮（竖大拇指图标），并确认点赞生效。

1. 确保已登录；未登录先走"登录流程"。
2. 在当前页面定位到用户指定的目标微博条目：
   - 若用户提供 URL：打开 URL 并确保已滚动到微博主体区域。
   - 若用户未提供 URL：点击左上角 Logo 旁的放大镜搜索入口，输入用户提供的关键词进行站内搜索，在搜索结果中打开并定位到目标微博条目后再继续；禁止在信息流里盲目滚动碰运气。
3. 必须目标条目的顶部区域在视野内，并且你能清楚看到发帖时间（位于发帖人昵称下方）后再继续。
4. 点击该条目标的发帖时间，进入该条微博的详情页面。
5. 若当前仍不是单条微博的详情视图，回到步骤 4，继续点击发帖时间进入详情页，直到确认已进入详情页为止。
6. 在详情页面中确认当前是单条微博的详情视图（能看到该条微博的发布人、时间、文案以及互动区）。未确认进入详情页前，禁止点赞。
7. 若微博条目过长导致点赞按钮不在当前屏幕，慢速向下滚动到该条微博详情的最下面区域（互动区/点赞区域）进入可见范围。
8. 在详情页面的互动区中寻找点赞按钮（通常是"竖大拇指"图标，旁边可能有点赞数）。
9. 点击点赞按钮。
10. 观察 UI 反馈确认成功（满足其一即可）：
    - 点赞图标样式/颜色变化（高亮）；
    - 点赞数 +1；
    - 出现"已赞/点赞成功"等提示。
11. 若点击后弹出登录提示或验证，回到"登录流程"并在登录后重试一次。

### 功能 A-2：收藏（登录后）

**目标**：通过目标微博条目右上角下拉菜单完成收藏，并确认收藏生效。

1. 确保已登录；未登录先走"登录流程"。
2. 定位到目标微博条目（同"点赞"的定位方法；若用户未给 URL，必须先使用左上角放大镜入口进行站内搜索定位目标条目）。
3. 必须目标条目的顶部区域在视野内，并且你能清楚看到发帖时间（位于发帖人昵称下方）后再继续。
4. 点击该条目标的发帖时间，进入该条微博的详情页面。
5. 若当前仍不是单条微博的详情视图，回到步骤 4，继续点击发帖时间进入详情页，直到确认已进入详情页为止。
6. 在详情页面中确认当前是单条微博的详情视图（能看到该条微博的发布人、时间、文案以及右上角的更多入口）。未确认进入详情页前，禁止收藏。
7. 确保你能清楚看到将要点击的入口后再继续。
8. 在该条微博发帖人昵称的右边，靠近条目的右上角边框处，寻找"V"字形（常见为"∨"形、V 字形）。
9. 点击该入口打开隐藏菜单/操作栏。
10. 在弹出的菜单中找到"收藏"并点击。
11. 观察 UI 反馈确认成功（满足其一即可）：
    - 菜单项变为"已收藏/取消收藏"；
    - 出现"收藏成功"等提示；
    - 相关图标状态变化。
12. 若找不到"收藏"，先确认你点击的是目标条目的菜单；必要时让详情页顶部区域完整可见后重试。

### 功能 A-3：发布文字微博（登录后）

**目标**：发布一条仅包含文字的微博，并确认发布成功。

1. 先收集要发布的文字内容：
   - 若用户已给出文本，直接使用；
   - 若未给出，向用户索取完整文本（包含换行、话题、@、表情符号等原样内容）。
2. 确保已登录；未登录先走"登录流程"。
3. 在微博主界面寻找发布入口：优先找右上角"铅笔在纸上写东西"的发布图标（或等价的"发微博/发布/写微博"入口）。
4. 点击发布入口，打开发布编辑框。
5. 将收集到的文字粘贴/输入到发布编辑框的文本框中（不要输入到左上角 Logo 旁的搜索框，也不要输入到信息流顶部的发布入口输入框以避免误触发），确保内容完整无截断。
6. 点击发送图标/发布按钮完成发布。
7. 观察 UI 反馈确认成功（满足其一即可）：
   - 出现"发布成功/发送成功"等提示；
   - 返回信息流后可在顶部看到刚发布的内容；
   - 进入个人主页可看到最新一条微博为刚发布内容。
8. 若发布过程中触发风控/验证，立即请求人工接管完成验证后再继续，不得反复快速重试导致更严格的限制。

### 失败与降级策略

- **图标找不到**：先滚动并等待页面加载；再通过"目标条目"的结构定位（头像/昵称/正文文本/时间戳）确认是否找对微博；仍失败则让用户提供目标微博 URL。
- **登录卡住**：必须说明卡在何处（验证码/短信/扫码/账号错误/权限不足），并请求用户继续人工处理；不要输出"已登录/已完成"之类未经验证的结论。
- **动作无反馈**：再次确认点击命中的是目标元素；必要时刷新页面后重试一次并重新验证状态。

## 模块 B：舆情分析

本模块把"微博热点话题/事件进展/舆情"转成可交付的内容包：**检索 → 选优 → 多视角补全 → 当事人核对 → 结构化汇总 →（可选）发帖文案**。

**触发本模块时，必须先读取 [weibo/CONSTRAINTS.md](weibo/CONSTRAINTS.md)。**

### 适用场景

- 某热点事件进展跟踪、舆情收集与归纳
- 围绕某话题的多视角观点整理（媒体/官方/当事人/专家/网友）
- 需要基于搜集结果生成一条或多条微博内容（信息型/观点型/提问型）

### 输入（缺失则按默认执行）

- **关键词/话题**：核心词 + 2-5 个同义/别称/相关词（若未给出则自动扩展）
- **时间窗**：默认近 7 天；若用户给相对时间，需写明换算后的日期区间与"检索截至日期（系统当前日期）"
- **地域/语言**：默认不限
- **关注对象**（可选）：当事人姓名/机构名/可能的微博账号；若未提供则在结果中反查候选账号
- **交付形式**：默认输出 1 份 Markdown 分析类交付物；若用户要"发帖/整理文案"，必须输出 2 份独立文档（分析/证据类文档 + 发帖文案文档），并按输出格式要求清晰展示文件边界

### 工作流

#### 1) 生成检索式（最多 3 组并行）

对每个主题生成 2-3 组互补检索式，覆盖：

- **事实核对**：关键词 + "通报/官方/警方/辟谣/澄清/公告/声明"
- **进展追踪**：关键词 + "最新/进展/后续/回应/通报/更新"
- **多视角补全**：关键词 + "媒体/专家/律师/医生/现场/目击/爆料/质疑"

优先使用能提高召回与可访问性的组合：

- `s.weibo.com/weibo?q=关键词`
- `s.weibo.com/weibo?q=关键词+进展`
- `s.weibo.com/weibo?q=关键词+通报`

必须补齐站内路径（覆盖事件进展、当事人、话题/超话）：

- 事件关键词 + 当事人名（用于把"事件"与"当事人发声"绑定）
- 事件关键词 + 话题符号（例如 `#事件词#`，用于定位话题页与讨论脉络）
- 当事人名 + "声明/回应/通报/澄清/辟谣/警方/法院/立案/判决/致歉"

#### 2) 站内实搜与进展跟踪（强制）

必须在微博站内完成以下动作，并在报告中留存可回溯的入口链接：

- 站内搜索：进入 `s.weibo.com` 的搜索结果页，至少翻看前 2 页或明显相关条目不足时继续扩展关键词
- 话题/超话：识别并进入相关话题页/超话页（若存在），记录话题名与可回溯入口链接，并抓取时间窗内的关键节点贴
- 当事人主页：进入当事人账号主页（优先使用 `weibo.com` 或 `m.weibo.cn` 可访问路径），查看置顶与时间窗内的相关发声；若有多个同名账号，必须基于认证主体/UID/主页信息做区分并说明理由
- 如遇登录/验证码/访问受限：立即切换为"请求用户协助"模式，向用户索要可打开的站内入口链接或截图/导出文本；不要改用站外信源补全信息

#### 2.1) 微博条目来源链接处理（强制）

- 输出给用户的参考来源时，必须围绕用户需求提供可核验链接；在大多数场景下，必须优先给出与结论**直接相关**的微博条目链接（尽量给原帖或关键转折节点帖），不要只给搜索页或主页链接。
- 涉及事实判断、时间线节点、争议点、当事人表态、官方通报时，必须附上对应微博条目链接；不可省略为"见搜索结果/见主页"。
- 获取单条微博链接时，必须严格按以下 GUI 步骤执行：
  1. 在当前微博帖子条目的右上方点击下标箭头 icon，展开更多选项后，点击"复制该微博地址"。
  2. 从 GUI 工具返回结果中读取 `clipboard_content`，该字段即当前帖子的链接；将该链接呈现给用户。
  3. 注意：不能直接复制浏览器地址栏网址作为帖子链接，必须严格按上述 1、2 步执行。
- 若因页面限制确实无法拿到单条微博链接，必须明确说明具体阻塞原因，并至少提供可回溯的替代入口链接（对应搜索结果/话题页/账号主页）与后续补链计划；不得无说明地省略来源链接。

#### 3) 采集候选内容

每条候选内容最少记录以下字段（缺失则标注未知）：

- 标题/摘要（可来自搜索摘要）
- 作者（昵称 + 主页链接/账号标识）
- 发布时间（以页面显示为准）
- 内容链接（尽量用可复现的 URL）
- 与主题的关系（1 句话）
- 抓取时间（系统当前日期）

#### 4) 去重与初筛

- URL 去重：同一链接只保留一条
- 近似去重：同一作者在短时间内的重复转发/同文多发，保留信息量更高者
- 初筛规则：明显无关、纯营销、无信息增量的内容移除（保留可疑但可能重要的条目进入"待核验"）

权威性优先规则（用于同类信息选优）：

- 优先级从高到低：官方通报/机构账号与认证主体 > 当事人账号 > 媒体与记者账号 > 专业人士账号 > 普通用户与目击叙述
- 对于"爆料/网传/据说"等表述，必须默认降级为"待核验"，除非能回溯到一手材料

#### 5) 价值评分（0–5 分）

对每条候选内容按以下维度评分，输出总分与原因（每维 1 句话）：

| 维度 | 5 分示例 | 0 分示例 |
| :--- | :--- | :--- |
| **相关性** | 直接描述事件关键事实/进展 | 仅蹭热度、无关内容 |
| **时效性** | 落在时间窗内且信息为最新节点 | 明显过时或时间不明 |
| **一手性** | 当事人/官方/现场目击/原始材料 | 二手转述且无出处 |
| **可核验性** | 有链接、截图/文件、原文引用可回溯 | 断言式结论、无依据 |
| **视角补充** | 提供不同立场/证据链的新增信息 | 与已有条目完全重复 |

默认选优策略：

- **核心条目**：总分 ≥ 18 且"一手性/可核验性"至少一项 ≥ 4
- **补充条目**：总分 14–17，用于补全时间线或多视角
- **待核验**：总分 < 14 或关键字段缺失，但可能影响判断（单列）

#### 6) 当事人核对（强制）

当主题涉及明确当事人/机构时：

- 先从候选内容中提取当事人账号线索（昵称、认证、主页 URL、UID）
- 再用站内检索补抓：`s.weibo.com/user?q=当事人名`、`s.weibo.com/weibo?q=当事人名+声明/回应/澄清`
- 输出"当事人账号清单"，并对其在时间窗内的相关发声做单独汇总

当用户明确要"当事人的话题/事件进展"时，必须额外输出"当事人相关话题与进展"小节：

- 当事人参与/被关联的核心话题（话题名、入口链接、与事件关系）
- 当事人关键发声的时间线（至少 3 个节点或尽可能完整）

#### 7) 多视角补全（强制）

对同一主题至少覆盖 3 类来源（能覆盖更多则更好）：

- 官方/机构通报
- 当事人/直接相关方
- 媒体报道/记者
- 行业专家/律师/医生等专业视角
- 普通网友/目击叙述（需标注不确定性）

如出现冲突信息：

- 并列陈述冲突点（谁说了什么、各自依据）
- 优先回溯一手来源；无法判定时降低结论语气，保留"待核验问题"

#### 8) 生成交付物（按需裁剪）

- 输出任何 Markdown 交付物时，先读取 [weibo/FORMAT.md](weibo/FORMAT.md)，使用其中的"命名规则"与模板，并确保文档名与问题高度一致（避免固定套路）。
- 必须保留：可回溯链接、抓取时间、过滤口径；不需要的章节直接删掉，只保留与问题相关的内容。

### 发帖/收藏/点赞（可选能力）

当用户要求"发帖/整理文案/收藏/点赞"时：

- 先读取 [weibo/FORMAT.md](weibo/FORMAT.md)，按其中"双份文档交付"的文件边界规则输出（两个独立 Markdown 代码块，且代码块外不附加解释性文字）。
- **发帖**：给出 1-3 条候选文案（类型必须贴合用户需求：信息型/观点型/提问型/澄清/征集线索等），并附：建议话题标签、素材引用（可回溯）、发布时机建议、风险提示（如信息未核验）。
- **收藏/点赞**：输出"执行清单表格"（动作、目标链接、理由、优先级）；若有可用 GUI 登录态，路由到模块 A 执行。

### 场景示例（触发与交付样式）

**示例 1：热点事件进展/舆情收集**

用户：收集"某事件"最近 48 小时微博进展，关注官方通报和当事人回应。

交付：按"摘要 → 时间线 → 多视角 → 当事人动态 → 关键条目清单 → 待核验"输出，并在摘要中写明检索截至日期。

**示例 2：热点话题了解与发帖**

用户：我想了解"某话题"的主要争议点，并帮我写一条信息型微博。

交付：先做多视角图谱与冲突点归纳，再提供信息型发帖文案（含来源链接与不确定性提示）。
