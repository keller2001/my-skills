# 非确定性任务处理

本文件覆盖"推荐攻略"、"推荐路线"、"推荐餐厅"等无唯一解需求的处理方法。

## 核心挑战

没有明确筛选要求，标准答案，容易导致思考过程过长、任务中断或死循环。

## 处理策略

### 1. 接受"满意解"而非"最优解"

- 设定合理的推荐数量（若用户不指定，则3-5 个；若不足 3 个，全部返回）
- 基于明确标准筛选
- 提供多样性（不同价位/类型/区域）
- 标注推荐理由，让用户自行选择

**示例**：
- 用户仅要求"推荐北京景点" → 输出 4 个景点，分别标注特色、适合人群、注意事项
- 用户仅要求"推荐上海餐厅" → 输出 5 家餐厅，覆盖不同菜系和价位

### 2. 建立筛选标准

从用户需求中提取显式标准与隐含标准，如无明确标准或缺少则采用通用指标，进行查缺补漏。

**通用指标**：
- 评分≥4.0
- 热度/人气排名前 N
- 距离≤10km
- 价格区间符合预算

**排序方式**：
- 按评分排序
- 按热度排序
- 按距离排序
- 综合评分（加权计算）

**示例**：
- "推荐杭州景点" → 筛选条件：评分≥4.0、距离杭州市区≤20km、开放中
- "推荐成都火锅" → 筛选条件：评分≥4.5、人均 100-200 元、营业中

### 3. 提供多样性

- 分类筛选（不同价位、不同类型、不同区域）
- 控制相似度（避免同一家连锁店的多家分店）
- 标注差异点（帮助用户区分）

**示例**：
- 推荐餐厅时：1 家川菜、1 家粤菜、1 家日料、1 家西餐
- 推荐景点时：1 个历史古迹、1 个自然风光、1 个现代商圈、1 个主题公园

### 4. 设置思考时限

- 简单推荐任务：思考≤3 轮搜索
- 中等推荐任务：思考≤5 轮搜索
- 复杂推荐任务：思考≤8 轮搜索
- 超过时限仍未确定，输出当前最佳结果

**超时处理**：
- 说明已进行多少轮搜索
- 列出当前找到的最佳选项
- 建议用户进一步细化需求
