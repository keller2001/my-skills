# Weibo Skill｜输出格式与交付模板（按需读取）

## 何时需要读取本文件

- 需要输出任何 Markdown 交付物（分析报告/事件跟踪报告/素材清单等）
- 用户要求“发帖/整理文案/可直接复制发布的内容”
- 用户要求“一次交付多份文档”，或对“最终回复格式”有明确约束

## 命名规则（让文档名与问题高度一致）

### 1) 先从问题里提取“主题短名”

- 优先沿用用户原话中的关键实体/话题（事件名、人物名、话题词、#话题#）。
- 用户给了多个并列主题时，用最能代表任务目标的那一个做主标题，其余放入副标题/范围说明。
- 用户没给明确主题时，用“用户意图 + 关键词”组成短名（例如“高校录取政策解读（微博舆情）”）。

### 2) 再选择“交付物类型”并拼成标题

交付物类型应直接匹配问题诉求，避免固定套路：

- 事件进展追踪：`事件进展跟踪`
- 舆情与观点：`微博舆情与观点整理`
- 当事人核对：`当事人动态核对与证据汇总`
- 发帖相关：`微博发帖文案`

### 3) H1 即“文件名”（用于清晰文件边界）

每份交付物在代码块内第一行必须为 H1，并使用书名号包裹文件名：

- `# 《{主题短名}｜{交付物类型}.md》`

如用户明确了时间窗/地域/关注对象，可在文件名中追加最小必要限定：

- `# 《{主题短名}｜{交付物类型}（{时间窗}）.md》`

## 只答所问（裁剪与措辞规则）

- 文档标题、章节标题尽量复用用户问法中的关键词，避免“通用报告/固定套路”命名。
- 只输出完成任务所必需的内容：用户没要的章节不写；不添加“流程说明/免责声明长文/通用科普”。
- 结论与文案措辞要匹配问题目标（例如“辟谣/澄清/征集线索/复盘/观点”），不要用统一模板口吻覆盖用户意图。

## 最终回复的“文件边界”规则

### 单份文档交付（默认）

- 最终回复只输出 **一个** Markdown 代码块。
- 代码块内包含完整文档内容，第一行为文件名 H1（见上文）。
- 代码块外不附加解释性文字。

### 双份文档交付（当用户要求发帖/整理文案时强制）

- 最终回复必须且只能包含 **两个独立的 Markdown 代码块**（不要把两个文档放在同一个代码块中；不要在代码块外附加解释性文字）。
- 第 1 个代码块：分析/证据类文档（报告/汇总/清单）。
- 第 2 个代码块：发帖文案文档（仅放可复制发布内容与发布要素，不放长篇分析）。
- 输出完成后做“交付自检”：确认代码块数量与每个代码块第一行的文件名 H1 均正确且与问题一致。

## 分析类交付模板（按需裁剪）

在不丢失“可回溯链接与抓取时间”的前提下，优先用“表格 + 编号引用”。只保留与问题相关的小节；不需要的章节直接删掉。

```markdown
# 《{主题短名}｜{交付物类型}.md》

## 信息快照
| 字段 | 内容 |
| :--- | :--- |
| 主题 | {用用户原话描述} |
| 时间窗 | {YYYY-MM-DD 至 YYYY-MM-DD；若未指定写默认并说明} |
| 检索截至 | {系统当前日期} |
| 关键词（含扩展） | {核心词 + 扩展词} |
| 覆盖范围 | 搜索 / 话题（含超话）/ 当事人主页 |
| 过滤口径 | {3-6 条要点} |

## 关键结论
- {结论 1}（证据：[S1] [S3]）
- {结论 2}（证据：[S2]）
- {结论 3}（证据：[S4]；不确定性：{一句话}）

## 关键信息与时间线（如问题要求“进展/后续/48小时”等）
| 时间 | 节点 | 最强证据（1–3 条） | 备注 |
| :--- | :--- | :--- | :--- |
| … | … | [S1] [S2] | … |

## 多视角观点（如问题要求“观点/争议/立场”）
| 视角 | 代表账号/来源 | 核心观点 | 证据/链接 | 不确定性 |
| :--- | :--- | :--- | :--- | :--- |
| 官方/机构 | … | … | [Sx] | … |
| 当事人/相关方 | … | … | [Sy] | … |
| 媒体/记者 | … | … | [Sz] | … |

## 当事人相关动态（如问题明确当事人/机构）
| 账号 | 认证主体/标识 | 主页链接 | 选择理由 |
| :--- | :--- | :--- | :--- |
| … | … | … | … |

| 时间 | 摘要 | 链接 |
| :--- | :--- | :--- |
| … | … | [Sx] |

## 关键争议与待核验（如存在冲突或证据缺口）
| 问题点 | 当前说法（并列） | 证据缺口 | 建议下一步（站内可执行） |
| :--- | :--- | :--- | :--- |
| … | A：…（[Sx]）/ B：…（[Sy]） | … | 用 `s.weibo.com/weibo?q=…` 定位一手材料 |

## 证据库与条目清单
| 编号 | 类型（核心/补充/待核验） | 时间 | 作者 | 摘要 | 评分（总分；关键维度） | 链接 | 抓取时间 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| S1 | 核心 | … | … | … | 21；一手性 5 / 可核验性 5 | {微博链接} | {系统当前日期} |

## 下一步（仅保留与问题相关的可执行动作）
- {站内可执行动作 1}
- {站内可执行动作 2}
```

## 发帖文案交付模板（仅放可复制发布内容）

必须让文案与问题目标一致：信息型/观点型/提问型/澄清/征集线索等。不要写固定开场白或通用套话；只保留对完成“发帖”必要的信息。

```markdown
# 《{主题短名}｜微博发帖文案.md》

## 风险提示（先看这个）
- {如：信息仍在核验中；引用条目见 [Sx]；避免定性措辞}

## 候选文案 1（{贴合用户要求的类型}）
正文：
{可直接复制的微博正文}

话题标签：
{#话题# #话题#（如适用）}

素材引用（可回溯）：
[S1] [S3]

配图/引用建议：
- {图片/截图/引用素材与对应链接}

发布时机建议：
- {一句话建议}

## 候选文案 2（可选；仅当用户需要多条或需要对照风格）
正文：
…
```
